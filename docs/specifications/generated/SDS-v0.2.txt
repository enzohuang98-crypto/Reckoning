1　文件目的
本文件目的在於描述「象棋 AI 分析講解軟體」之軟體設計規格，作為後續 MVP 開發、Claude Code 自動開發任務拆解、程式碼審查、測試驗收與產品溝通之依據。
本文件涵蓋系統架構、設計方法、開發工具、模組劃分、資料結構、使用者介面、例外處理、外部整合、備份回復、安全機制與共用模組等內容。
本產品的核心不是取代引擎算棋，而是把 Pikafish 的分析結果轉換成棋手能理解、能追問、能儲存與複習的長篇教練式分析。
2　工作說明
2.1　設計方法與工具
2.1.1　系統設計方法與架構
本系統採 Windows 桌面 App 架構，不採 Web SaaS，不需使用者登入，不提供雲端同步，也不建置自有後端伺服器。
第一版採本機優先設計：Pikafish 於使用者電腦本機執行，軟體透過 UCI engine adapter 與引擎溝通；Claude、OpenAI、Gemini 等 AI Provider 則透過使用者自行填入的 API Key 進行雲端模型呼叫。
重要名詞限制：本文件不將 Pikafish 稱為雲端 API。Pikafish 在本設計中被視為「本機 UCI 象棋引擎」。若程式內出現 API 字樣，僅指本軟體內部封裝之 adapter interface，不代表 Pikafish 有官方 HTTP API。
系統主要分為六層：使用者介面層、應用服務層、引擎整合層、AI Provider 整合層、本機資料儲存層、安裝與發佈層。

本專案採重複式開發（Iteration）為主，並結合由上而下與由下而上的設計方式。

2.1.6　系統分析與設計方法
本專案採物件導向與元件化設計。對外部不穩定部分，例如 Pikafish UCI output 與不同 AI Provider API，使用 adapter interface 隔離；對核心資料流，例如 EngineAnalysis 與 MistakeBookEntry，使用固定 TypeScript type 作為契約。
核心設計原則：Pikafish 負責棋力判斷，LLM 只負責解釋結構化引擎資料。任何 AI 解釋不得自行發明不在引擎資料中的戰術、殺法、勝勢或計畫。

2.6　資料結構設計（核心型別契約）
本系統的核心資料結構必須先固定，因為 AI 解釋、錯題本、成本估算與安全規則都依賴穩定資料契約。以下型別為 v0.2 之後的唯一真相來源；MoveComparisonService、EngineOutputParser、PikafishAdapter、StorageService、UI 與 IPC 都必須使用同一組型別。
2.6.1　分數型別 EngineScore
Pikafish 在一般局面回傳 centipawn（score cp N），在殺棋或接近殺棋時回傳 mate score（score mate N）。為避免 UI 與 AI 解釋失去語義，分數不得只存裸數字。
type ScoreSource =
  | &quot;root_analysis&quot;
  | &quot;candidate_move&quot;
  | &quot;separate_engine_call&quot;;
type EngineScore =
  | {
      type: &quot;cp&quot;;
      cp: number;            // Pikafish 原始 centipawn
      value: number;         // 兵/卒單位 (cp / 100)
      comparableValue: number; // 給 MoveComparisonService 比較用
      raw: string;           // 原始 UCI 字串，僅供 debug
      displayText: string;   // UI 顯示文字，如 +1.20
      wasInverted: boolean;  // 是否經過視角反轉
      source: ScoreSource;
    }
  | {
      type: &quot;mate&quot;;
      mateIn: number;        // mate 距離；負數表示自己將被將死
      comparableValue: number;
      raw: string;
      displayText: string;   // 如「殺 3」「被殺 2」「已被將死」
      isTerminalMate: boolean; // mate 0 終局狀態
      wasInverted: boolean;
      source: ScoreSource;
    };

2.13　MoveComparisonService 錯誤分級規則
2.13.1　設計目的
MoveComparisonService 比較使用者走法與 Pikafish 推薦走法之間的評估差距，並轉換為棋手能理解的錯誤等級。本系統不得只顯示「你錯了」，而必須說明：
使用者走法造成多少評估損失。
該損失屬於可接受、緩手、明顯錯誤、嚴重錯誤，或重大敗著。
錯誤等級只是產品輔助判斷，不等於絕對棋理真理。
若引擎深度不足或資料缺失，必須標示可信度不足。
2.13.2　評估分數正規化與 scoreDifference 計算
Pikafish 原始評估分數以紅方角度呈現：分數越高代表紅方越好。比較前必須依 sideToMove 正規化成「對目前行棋方有利」的方向。normalizeScore() 只接受有效數字，不處理 null。
function normalizeScore(rawScore: number, sideToMove: &quot;red&quot; | &quot;black&quot;): number {
  return sideToMove === &quot;red&quot; ? rawScore : -rawScore;
}
evaluationAfterBestMove 與 evaluationAfterUserMove 皆可能為 number | null。若任一為 null，scoreDifference 必須為 null，不得嘗試計算，也不得用 0 補值。
const scoreDifference: number | null =
  evaluationAfterBestMove === null || evaluationAfterUserMove === null
    ? null
    : normalizeScore(evaluationAfterBestMove, sideToMove) -
      normalizeScore(evaluationAfterUserMove, sideToMove);
若行棋方為黑方，必須反轉分數方向，否則會把「對紅方有利」誤判為「對黑方有利」。
2.13.3　MistakeLevel 明確區間表
本系統使用半開區間 [a, b)：包含左邊界，不包含右邊界。例如 [0.31, 0.81) 代表 0.31 &lt;= scoreDifference &lt; 0.81。

2.13.4　為什麼不用 turning_point
第一版不使用 turning_point。單純的 scoreDifference &gt; 3.00 只代表使用者走法造成重大損失，不能證明該步是整盤棋的勝負轉折點。例如局面原本已是黑方大優，使用者再走更差的一步，分數會繼續惡化，但這只是「輸得更嚴重」，不是「轉折」。因此第一版使用 major_blunder，語義上只表示「重大損失」。
未來若要偵測真正的 turning point，應新增獨立欄位（例如 isTurningPointCandidate: boolean），並至少需要 evaluationBefore、evaluationAfterUserMove、evaluationAfterBestMove，以及分數是否跨越指定臨界值。此功能不列入第一版 MVP。
2.13.5　classifyMistakeLevel 實作（純函式）
function classifyMistakeLevel(scoreDifference: number | null): MistakeLevel {
  if (
    scoreDifference === null ||
    Number.isNaN(scoreDifference) ||
    !Number.isFinite(scoreDifference)
  ) {
    return &quot;unknown&quot;;
  }
  if (scoreDifference &lt; 0.31) return &quot;acceptable_or_tiny_inaccuracy&quot;;
  if (scoreDifference &lt; 0.81) return &quot;inaccuracy&quot;;
  if (scoreDifference &lt; 1.51) return &quot;mistake&quot;;
  if (scoreDifference &lt; 3.01) return &quot;serious_mistake&quot;;
  return &quot;major_blunder&quot;;
}
2.13.6　confidence 計算規則
以下每一項都算一個 uncertainty reason：
depth 缺失。
depth 低於使用者設定的最低深度。
candidateMoves 少於 2 個。
principalVariation 為空。
evaluationAfterUserMove 缺失。
evaluationAfterBestMove 缺失。
scoreDifference 無法計算。
confidence 映射：0 個 → high；1 個 → medium；2 個以上 → low。
強制 low：以下任一情況即使只有 1 個 reason 也必須直接判為 low：evaluationAfterUserMove 缺失、evaluationAfterBestMove 缺失、scoreDifference 無法計算、engineBestMove 缺失、或 principalVariation 為空且 candidateMoves 少於 2 個。
2.13.7　Claude Code 實作限制
不得自行修改 mistakeLevel 閾值。
不得使用 turning_point。
不得用 LLM 判斷錯誤等級；只能由結構化引擎資料計算。
不得把 scoreDifference &lt; 0.00 判定為錯誤。
必須使用本節定義的半開區間；分級邏輯必須寫成可單元測試的純函式。
分類必須使用原始 scoreDifference，不得使用 UI 四捨五入後數值。
若 scoreDifference 為 null、NaN 或 Infinity，必須回傳 unknown。
若行棋方為黑方，必須先做評估分數正規化。
confidence 必須依本節規則計算，不得自行猜測。
2.14　EngineOutputParser：EngineScore 與 mate score 處理
2.14.1　設計目的
Pikafish 一般局面回傳 score cp N，殺棋或接近殺棋回傳 score mate N。EngineOutputParser 必須同時保留：(1) 可供 MoveComparisonService 計算的數值；(2) 可供 UI 顯示的原始 score 類型；(3) 可供 AI 解釋使用的 mate 資訊。不得只把 score mate 3 轉成 29997 後丟掉原始語義，也不得讓 Infinity / -Infinity 進入 EngineAnalysis。
2.14.2　convertCpScore
function convertCpScore(
  cp: number,
  raw: string,
  source: ScoreSource = &quot;root_analysis&quot;
): EngineScore {
  const value = cp / 100;
  return {
    type: &quot;cp&quot;,
    cp,
    value,
    comparableValue: value,
    raw,
    displayText: value &gt;= 0 ? `+${value.toFixed(2)}` : value.toFixed(2),
    wasInverted: false,
    source,
  };
}
2.14.3　convertMateScore（含 mate 0 terminal case）
一般 parser 階段，score mate 0 代表目前分析局面的行棋方已被將死，因此 comparableValue = -MATE_SCORE，displayText = 「已被將死」。注意：此為「未反轉」語義；第二次分析後的反轉由 PikafishAdapter 的 invertEngineScore() 負責（見 §2.15）。
const MATE_SCORE = 30000;
function convertMateScore(
  mateIn: number,
  raw: string,
  source: ScoreSource = &quot;root_analysis&quot;
): EngineScore {
  if (mateIn === 0) {
    return {
      type: &quot;mate&quot;,
      mateIn: 0,
      comparableValue: -MATE_SCORE,
      raw,
      displayText: &quot;已被將死&quot;,
      isTerminalMate: true,
      wasInverted: false,
      source,
    };
  }
  const sign = mateIn &gt; 0 ? 1 : -1;
  const distancePenalty = Math.min(Math.abs(mateIn), 100);
  const comparableValue = sign * (MATE_SCORE - distancePenalty);
  return {
    type: &quot;mate&quot;,
    mateIn,
    comparableValue,
    raw,
    displayText: mateIn &gt; 0 ? `殺 ${mateIn}` : `被殺 ${Math.abs(mateIn)}`,
    isTerminalMate: false,
    wasInverted: false,
    source,
  };
}

2.15　PikafishAdapter：userMove 分析與視角反轉
2.15.1　設計目的
系統需取得 scoreAfterUserMove 與 scoreAfterBestMove。若 userMove 不在 root analysis 的 candidateMoves 中，系統必須對 userMove 後的局面進行第二次 Pikafish 分析，否則 MoveComparisonService 只能回傳 unknown，降低產品核心價值。
2.15.2　雙階段分析流程
使用者提交 userMove
  → PikafishAdapter 分析原局面 (root analysis)
  → 取得 bestMove、scoreAfterBestMove、candidateMoves
  → 檢查 userMove 是否在 candidateMoves
     是 → 使用 candidate.score 作為 scoreAfterUserMove
          (source: candidate_move，不取負號)
     否 → 套用 userMove 產生新局面 → 第二次分析
          (source: separate_engine_call，必須取負號)
  → 第二次分析失敗 → scoreAfterUserMove = null，加入 uncertainty reason
  → 建立 EngineAnalysis
2.15.3　UCI score 視角規則
UCI engine 的 score 相對於該分析局面的行棋方。第一次分析的局面是原局面；第二次分析的局面是 userMove 後的局面，此時行棋方已換邊，回傳的 score 是對方視角。要與 scoreAfterBestMove 比較，必須把第二次分析的 score 取負號，轉回原局面行棋方視角。

2.15.4　invertEngineScore（含 mate 0 修正）
mate 0 在反轉後語義改變：第二次分析回傳 mate 0 代表 userMove 後輪到的對方已被將死，即 userMove 是殺棋著法。從對方視角是 -MATE_SCORE，反轉回原局面行棋方視角後必須變成 +MATE_SCORE，顯示「殺棋（終局）」，不得仍顯示「已被將死」。
function invertEngineScore(score: EngineScore): EngineScore {
  if (score.type === &quot;cp&quot;) {
    const invertedCp = -score.cp;
    const value = invertedCp / 100;
    return {
      type: &quot;cp&quot;,
      cp: invertedCp,
      value,
      comparableValue: value,
      raw: score.raw,
      displayText: value &gt;= 0 ? `+${value.toFixed(2)}` : value.toFixed(2),
      wasInverted: true,
      source: &quot;separate_engine_call&quot;,
    };
  }
  const invertedMateIn = -score.mateIn;
  if (invertedMateIn === 0) {
    return {
      type: &quot;mate&quot;,
      mateIn: 0,
      comparableValue: MATE_SCORE,   // 反轉後為正：使用者剛將死對方
      raw: score.raw,
      displayText: &quot;殺棋（終局）&quot;,
      isTerminalMate: true,
      wasInverted: true,
      source: &quot;separate_engine_call&quot;,
    };
  }
  const sign = invertedMateIn &gt; 0 ? 1 : -1;
  const distancePenalty = Math.min(Math.abs(invertedMateIn), 100);
  const comparableValue = sign * (MATE_SCORE - distancePenalty);
  return {
    type: &quot;mate&quot;,
    mateIn: invertedMateIn,
    comparableValue,
    raw: score.raw,
    displayText:
      invertedMateIn &gt; 0
        ? `殺 ${invertedMateIn}`
        : `被殺 ${Math.abs(invertedMateIn)}`,
    isTerminalMate: false,
    wasInverted: true,
    source: &quot;separate_engine_call&quot;,
  };
}
2.15.5　raw 欄位使用規則
反轉後 raw 仍是原始 UCI 字串，其正負號與反轉後的數值相反。為避免 AI 解釋方向錯誤，raw 只能供 debug 與追蹤，不得被 PromptBuilder、MoveComparisonService、UI 顯示或 mistakeLevel 分級使用。
UI 必須使用 score.displayText。PromptBuilder 只能使用 score.type、score.displayText、score.comparableValue 以及（type === &quot;mate&quot; 時）score.mateIn，且不得把 raw 放進送給 LLM 的 prompt。

2.15.7　第二次分析失敗處理

2.15.8　Claude Code 實作限制
candidate_move 來源不取負號；separate_engine_call 來源必須取負號。
不得直接把第二次分析的 raw score 存入 scoreAfterUserMove。

2.16　Electron IPC 架構
2.16.1　設計原則
Electron 的 main process 與 renderer process 不能直接互相呼叫，必須透過 IPC channel。PikafishAdapter、AIProvider、SecretStore、AnalysisSessionStore 都在 main process；BoardEditor、AnalysisPanel 等 UI 在 renderer。EngineAnalysis 與 MoveComparisonResult 的真相來源必須在 main process；renderer 只接收顯示用資料與 analysisId，不得把分析資料再傳回 main 作為 AI 解釋依據。
安全邊界：API Key 只能在 main process 或安全設定層取得，不得暴露給 renderer 任意讀取。Renderer 不得直接呼叫 provider API；provider adapter 不得直接操作 renderer UI。PromptBuilder 必須在 main 或受控 service 層執行，不得散落在 UI component。

analysisId 必須由 main process 生成（renderer 只生成 requestId）。必須在 EngineAnalysis 與 MoveComparisonResult 都建立完成後產生，並先成功存入 AnalysisSessionStore，再發送 engine:analysis-result。順序不得顛倒。save() 必須包 try/catch；失敗時送 engine:analysis-error（code: &quot;session_store_failed&quot;），不得讓 renderer 永遠等待。
const analysisId = crypto.randomUUID();
const now = Date.now();
const session: AnalysisSession = {
  analysisId,
  requestId: payload.requestId,
  createdAt: new Date(now).toISOString(),
  expiresAt: new Date(now + DEFAULT_ANALYSIS_SESSION_TTL_MS).toISOString(),
  positionFen: payload.positionFen,
  userMove: payload.userMove,
  engineAnalysis,
  moveComparison,

try {
  await analysisSessionStore.save(session);
  event.reply(&quot;engine:analysis-result&quot;, {
    requestId: payload.requestId,
    analysisId,
    engineAnalysis,
    moveComparison,
  });
} catch (error) {
  event.reply(&quot;engine:analysis-error&quot;, {
    requestId: payload.requestId,

2.16.5　engine:analysis-cancel 取消機制
Engine 取消與 AI streaming 取消（§2.17）採對稱設計：以 requestId 管理進行中的分析。每個進行中的分析在 main process 註冊一個 handle，內含 AbortController 與目前使用的 Pikafish 子程序參照。
const activeEngineAnalyses = new Map&lt;string, EngineAnalysisHandle&gt;();
type EngineAnalysisHandle = {
  requestId: string;
  controller: AbortController;
  // 目前進行中的階段：root 或 userMove 二次分析
  phase: &quot;root_analysis&quot; | &quot;user_move_analysis&quot;;
  // 對目前使用中的 Pikafish 子程序送出 UCI 指令
  sendStop: () =&gt; void;        // 寫入 &quot;stop&quot; 到 engine stdin
  killEngine: () =&gt; void;      // 寬限期後強制終止子程序
};
UCI 協議規定：收到 stop 後，引擎仍會送出一個最終 bestmove（每個 go 都必須對應一個 bestmove）。因此取消流程是：
依 requestId 找到 handle。
呼叫 controller.abort()，讓雙階段流程（不論在 root 或 userMove 階段）停止往下走。
對目前使用中的子程序送出 UCI &quot;stop&quot;。
等待引擎送出最終 bestmove 後丟棄；若在寬限期（例如 500ms）內未回應，呼叫 killEngine() 強制終止子程序避免卡死。
不得發送 engine:analysis-result；改送 engine:analysis-error（code: &quot;cancelled&quot;）。
從 activeEngineAnalyses 移除該 requestId。
兩個階段的取消行為一致：第二次（userMove）分析檢查同一個 AbortSignal，若 abort 在二次分析期間觸發，對二次分析的子程序送 stop。
function handleEngineCancel(payload: { requestId: string }): void {
  const handle = activeEngineAnalyses.get(payload.requestId);
  if (!handle) return;
  handle.controller.abort();
  handle.sendStop();
  // 在主分析迴圈中，abort 觸發後丟出 AbortError，
  // catch 分支送出 engine:analysis-error code &quot;cancelled&quot;
  activeEngineAnalyses.delete(payload.requestId);
}
主分析迴圈（root + userMove）必須以 try/catch/finally 包住，AbortError 對應 code &quot;cancelled&quot;，其他例外對應對應的 engine_* 錯誤碼；finally 必須從 activeEngineAnalyses 移除 requestId，避免 handle 洩漏。
2.17　AI 解釋 Streaming IPC
2.17.1　設計決策
ai:generate-explanation 第一版必須採 streaming IPC，不採單次 request/response。長篇分析可能需數秒到數十秒；streaming 讓使用者即時看到文字逐段出現，也較易支援取消。即使某 provider SDK 尚未支援真 streaming，adapter 仍須包裝成相同 streaming 介面（先等完整回應，再以單一 text_delta chunk 回傳，最後送 done），UI 與 IPC 層不得因此改回單次 response 架構。

2.17.5　Main process streaming loop（最終版）
重點：(1) accumulatedText 累積完整文字（done chunk 不含 finalText）；(2) done 分支先設 completedNormally=true 再 reply，然後 break；(3) abort 檢查放在 chunk 處理「之後」，避免 done 與 cancelled 同時送出的 race condition；(4) catch 僅在 !completedNormally 時送 error；(5) finally 清除 activeExplanationRequests 避免 memory leak。
const activeExplanationRequests = new Map&lt;string, AbortController&gt;();
async function handleGenerateExplanationStart(
  event: Electron.IpcMainEvent,
  payload: GenerateExplanationStartPayload
): Promise&lt;void&gt; {
  const { requestId } = payload;
  const controller = new AbortController();
  activeExplanationRequests.set(requestId, controller);
  let accumulatedText = &quot;&quot;;
  let completedNormally = false;
  try {
    const provider = getAIProvider(payload.provider);
    const request = await buildAIExplanationRequest(payload, {
      secretStore, promptBuilder, modelRegistry,
      settingsService, analysisSessionStore,
    });
    for await (const chunk of provider.generateExplanationStream(
      request, controller.signal
    )) {
      if (chunk.type === &quot;text_delta&quot;) {
        accumulatedText += chunk.deltaText;
        event.reply(&quot;ai:generate-explanation:chunk&quot;, {
          requestId, deltaText: chunk.deltaText,
        });
      }
      if (chunk.type === &quot;done&quot;) {
        completedNormally = true;
        event.reply(&quot;ai:generate-explanation:done&quot;, {
          requestId,
          finalText: accumulatedText,
          usage: chunk.usage,
          estimatedCostUsd: chunk.estimatedCostUsd ?? null,
        });
        break;
      }
      if (controller.signal.aborted) {
        throw new DOMException(&quot;Request cancelled&quot;, &quot;AbortError&quot;);
      }
    }
  } catch (error) {
    if (!completedNormally) {
      const errorPayload = mapStreamingErrorToPayload(requestId, error);
      event.reply(&quot;ai:generate-explanation:error&quot;, errorPayload);
    }
  } finally {
    activeExplanationRequests.delete(requestId);
  }
}

2.17.7　Renderer 規則
產生唯一 requestId，發送 start。

2.17.8　getAIProvider 合約
getAIProvider() 只依 provider 名稱回傳對應 adapter，不讀 API key、不呼叫 PromptBuilder、不讀寫 AppSettings、不碰 UI、不決定 fallback model。
function getAIProvider(providerName: AIProviderName): AIProvider {
  switch (providerName) {
    case &quot;anthropic&quot;: return anthropicProvider;
    case &quot;openai&quot;:    return openAIProvider;
    case &quot;gemini&quot;:    return geminiProvider;

2.17.9　buildAIExplanationRequest 合約
buildAIExplanationRequest() 是 SecretStore、PromptBuilder、ModelRegistry、AnalysisSessionStore 的集中銜接點。IPC handler 不得自己組 prompt 或讀 API key。
type AIExplanationRequest = {
  provider: AIProviderName;
  model: string;
  apiKey: string;
  prompt: string;
  metadata: {
    requestId: string;
    analysisId: string;
    userLevel: &quot;basic&quot; | &quot;intermediate&quot; | &quot;advanced&quot;;
    explanationStyle: &quot;long_analytical&quot;;
  };
};
async function buildAIExplanationRequest(
  payload: GenerateExplanationStartPayload,
  deps: {
    secretStore: SecretStore;
    promptBuilder: PromptBuilder;
    modelRegistry: ModelRegistry;
    settingsService: SettingsService;
    analysisSessionStore: AnalysisSessionStore;
  }
): Promise&lt;AIExplanationRequest&gt; {
  const modelConfig = deps.modelRegistry.getModel(
    payload.provider, payload.model
  ); // 不存在則丟 UnsupportedModelError
  const apiKey = await deps.secretStore.getApiKey(payload.provider);
  if (!apiKey) throw new MissingApiKeyError(payload.provider);
  const session = await deps.analysisSessionStore.get(payload.analysisId);
  if (!session) throw new AnalysisSessionNotFoundError(payload.analysisId);
  const prompt = deps.promptBuilder.buildExplanationPrompt({
    engineAnalysis: session.engineAnalysis,
    moveComparison: session.moveComparison,
    userLevel: payload.userLevel,
    explanationStyle: payload.explanationStyle,
  });
  return {
    provider: payload.provider,
    model: modelConfig.model,
    apiKey,
    prompt,
    metadata: {
      requestId: payload.requestId,
      analysisId: payload.analysisId,
      userLevel: payload.userLevel,
      explanationStyle: payload.explanationStyle,
    },
  };
}
PromptBuilder 只讀取 EngineAnalysis、MoveComparisonResult、userLevel、explanationStyle；不得讀取 API key、raw UCI score、renderer UI state、provider SDK response 或 SecretStore。SecretStore 缺 key 時丟 MissingApiKeyError，不得用空字串或 placeholder 繼續呼叫。
2.17.10　Claude Code 實作限制
不得使用單次 ipcMain.handle 作為最終設計；必須 chunk/done/error/cancel 齊全。
abort 檢查必須放在 chunk 處理之後；done 分支先設 completedNormally=true 再 reply，並 break。
catch 僅在 !completedNormally 時送 error；finally 必須清除 activeExplanationRequests。
userLevel 必須來自 AppSettings.userLevel，fallback &quot;intermediate&quot;；不得在 AnalysisPanel 內寫死。
cancel 不得只改 UI 狀態，必須實際呼叫 controller.abort()。
buildAIExplanationRequest() 是唯一組裝 prompt 與注入 API key 的入口；IPC handler 不得散落 prompt 字串或把 key 寫進 adapter 全域狀態。
PromptBuilder 不得使用 EngineScore.raw。

2.18.3　第一版實作（記憶體 Map）與 TTL
const DEFAULT_ANALYSIS_SESSION_TTL_MS = 2 * 60 * 60 * 1000; // 2 hours
class InMemoryAnalysisSessionStore implements AnalysisSessionStore {
  private sessions = new Map&lt;string, AnalysisSession&gt;();
  async save(session: AnalysisSession): Promise&lt;void&gt; {
    await this.clearExpiredSessions();   // save() 前先清理
    this.sessions.set(session.analysisId, session);
  }
  async get(analysisId: string): Promise&lt;AnalysisSession | null&gt; {
    const session = this.sessions.get(analysisId);
    if (!session) return null;
    if (Date.now() &gt; Date.parse(session.expiresAt)) {
      this.sessions.delete(analysisId);
      return null;
    }
    return session;
  }
  async delete(analysisId: string): Promise&lt;void&gt; {
    this.sessions.delete(analysisId);
  }
  async clearExpiredSessions(): Promise&lt;void&gt; {
    const now = Date.now();
    for (const [id, s] of this.sessions.entries()) {
      if (now &gt; Date.parse(s.expiresAt)) this.sessions.delete(id);
    }
  }
}
2.18.4　清理觸發（兩層）
第一版採兩層清理策略：(1) 每次 save() 前呼叫 clearExpiredSessions()；(2) app 啟動後設定定時清理。

第一版建議使用本機 SQLite 儲存錯題本、看不懂局面、歷史分析與非敏感設定；API Key 應使用 OS secret store，不應直接存在 SQLite 明文欄位。本資料庫選型仍屬待確認（Q2）。

原則：錯誤要可理解、可復原，不可讓使用者以為不完整分析是確定結論。

處理方式
使用者提示

2.10　備份與回復機制
第一版不使用自有雲端伺服器，備份與回復以本機匯出/匯入為主。錯題本與看不懂局面以 JSON 或 SQLite backup 匯出，匯入時需保留 schemaVersion 並避免重複資料。歷史 AI 解釋可隨錯題本一起匯出，回復後可離線查看已生成文字、不重新計費。API Key 不納入一般匯出，新電腦需重新輸入。模型價格表隨 App 附帶並可手動更新，需顯示 lastUpdated。
2.11　安全機制
主要安全風險不是傳統資料庫攻擊，而是 API Key 洩漏、AI 解釋幻覺、外部成本不可控，以及 Claude Code 自動開發時不小心提交秘密資訊。
風險
防護設計
API Key 洩漏
輸入欄位遮蔽；不可寫入程式碼、不可 console.log、不可上傳到自有伺服器；優先使用 OS secret store。
模型亂判棋
PromptBuilder 要求模型只根據 EngineAnalysis；缺資料時顯示可信度不足；Pikafish 是棋力來源。
成本失控
每次解釋前顯示預估 token 與估算價格；價格由 model_pricing.json 管理。
Claude Code 亂改
使用 CLAUDE.md、skills、protected files、build/test 驗收；高風險任務不得無人看守。
資料遺失
錯題本支援匯出；本機儲存失敗時保留畫面內容。
Prompt injection
使用者輸入與引擎資料僅作分析資料，不得被視為系統指令；系統 prompt 優先。
Log 洩漏
logger 自動遮蔽 apiKey、Authorization header、secret token。

  No API key may be hardcoded, logged, committed,
